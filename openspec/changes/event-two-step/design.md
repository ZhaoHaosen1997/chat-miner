## Context

v1.18.0 事件探测将 Phase 1（Python 尖峰检测→固定窗口）和 Phase 2（AI 逐窗口分析→入库）捆绑在一个 `asyncio.create_task` 中，且在 Phase 2 前全局 `DELETE` 旧事件。这导致两个严重问题：

1. **数据安全**：任务崩溃/取消 → 旧事件永久丢失 + 新事件不完整
2. **切分质量**：固定 200 条消息窗口无视对话自然边界，一个事件可能被切碎或与无关消息混在一起

v1.18.1 的核心策略：**持久化中间状态 + 自适应事件边界 + 队列串行**。这正是项目"Python 做统计，AI 做总结"原则在事件探测中的完整贯彻。

## Goals / Non-Goals

**Goals:**
- Phase 1 检测结果持久化到 `event_windows` 表，不依赖内存状态
- 自适应时间间隙切分替换固定窗口，实现一个事件组 = 一段自然对话 = 一次 AI 调用
- AI 分析从检测任务中独立出来，队列串行执行，逐个入库
- 窗口级别数据安全：只删除该窗口关联的旧事件，不动其他数据
- 前端两步交互：检测→列表展示→逐条/批量分析→自动刷新

**Non-Goals:**
- 自动/定时事件检测（仍为纯手动触发）
- 事件重要性评分
- 实时增量检测（消息导入时不触发）
- 修改报告生成流程（事件仅在展示层引用）

## Decisions

### 1. 自适应切分算法：时间间隙 + 动态阈值

**选择**：在合并后的尖峰时段内，计算相邻消息的时间间隙，在显著间隙处切分。

**算法**：
```
1. 对尖峰时段内所有消息按时间排序
2. 计算相邻消息间的时间间隔 gaps = [g1, g2, ..., gn]（单位：分钟）
3. 动态阈值 = max(15分钟, min(P75(gaps) × 1.5, 60分钟))
   - 绝对下界 15 分钟：防止在密集对话中过度切分
   - 相对上界 60 分钟：防止阈值过大导致从不切分
   - P75 × 1.5：自适应群聊节奏，活跃群阈值低，安静群阈值高
4. 在 gap ≥ threshold 的位置切分
5. 后处理：
   - 消息数 < 10 的段合并到相邻段（选间隙小的那边）
   - 消息数 > 500 的段在次优间隙处再切一刀（保证每组 ≥ 50 条）
```

**替代方案**：
- 固定窗口（当前方案）：已被证明有问题
- 固定时间窗口（如 1 小时）：不同群聊密度差异巨大
- Embedding 语义切分：引入新依赖，计算成本高，群聊话题切换模糊
- 纯基于参与者变化切分：不够可靠，同一事件参与者也可能变化

**理由**：时间间隙是群聊事件边界最自然的信号——"大家不聊了"就是事件结束。动态阈值适应不同群的聊天节奏。

### 2. AI 输出格式：单个事件对象，非数组

**选择**：Prompt 要求 AI 返回单个事件对象 `{title, description, event_type, ...}` 或 `null`。

**替代方案**：
- 保持数组格式（当前方案）：一个窗口可能返回 0-N 个事件，增加解析复杂度和不确定性
- 要求固定返回 1 个事件：即使没有事件也强制作答，降低质量

**理由**：自适应切分后，一个事件组 = 一段自然对话，大概率是 0 或 1 个事件。AI 从"在一堆消息里找所有事件"变为"这段对话是不是一个事件？"，判断比搜索更准确。

### 3. 队列串行 vs 并发

**选择**：队列串行（一个接一个分析），放弃 v1.18.0 的 asyncio.Semaphore(3) 并发。

**替代方案**：
- 保持 3 并发：用户反馈"已导致大模型返回不可用"
- 批量 AI 调用（一次 prompt 分析多个窗口）：prompt 过长，质量下降

**理由**：串行处理对 API 友好，失败隔离好（一个窗口失败不影响后续），用户可随时取消。速度损失可接受——串行 5 个窗口约 2-3 分钟，比并发被限流后不断重试更快。

### 4. 数据模型：event_windows 表 + events.window_id

**选择**：新增 `event_windows` 表存储检测结果，`events` 表增加 `window_id` 列关联来源窗口。

```
event_windows                    events
┌──────────────────────┐        ┌──────────────────────┐
│ id (PK)              │◄───────│ window_id (FK)       │
│ group_id (FK)        │        │ ... (existing cols)  │
│ start_time           │        └──────────────────────┘
│ end_time             │
│ message_count        │
│ status               │  状态机：
│ event_count          │  pending → analyzing → analyzed
│ summary_json         │                     → empty
│ event_id (FK→events) │  analyzing → pending (retry)
│ created_at           │
│ analyzed_at          │
└──────────────────────┘
```

**替代方案**：
- 复用 `events` 表加 `status` 列区分：语义混乱，已分析/待分析是不同实体
- 纯内存状态（当前方案）：任务崩溃全丢

**理由**：独立表语义清晰，支持独立的 CRUD 和状态机。`window_id` 双向关联支持"重新分析单个窗口时精确删除旧事件"。

### 5. API 设计

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/groups/{id}/events/detect` | POST | Phase 1 检测，返回 `{windows: [...], count: N}` |
| `/api/groups/{id}/events/windows` | GET | 获取事件窗口列表，支持 `?status=pending` 筛选 |
| `/api/groups/{id}/events/windows/{wid}/analyze` | POST | 分析单个窗口 |
| `/api/groups/{id}/events/windows/analyze-all` | POST | 一键分析所有待处理窗口（后台串行队列） |
| `/api/groups/{id}/events` | GET | 获取已分析事件列表（保持不变） |

`POST /detect` 从返回 `{task_id}` 改为返回 `{windows, count}` —— **BREAKING**，但 v1.18.0 才发布不久，影响可控。

### 6. 前端架构

```
Events.vue
├─ 检测区（日期选择 + "开始分析"按钮）
├─ 窗口列表区（检测完成后显示）
│   ├─ 月份分组
│   ├─ 窗口卡片 × N（摘要 + "分析"按钮 + 状态徽标）
│   └─ "一键分析"按钮（仅在有待处理窗口时显示）
└─ 事件时间轴（已有 EventTimeline，已分析事件自动出现）
```

与日报告/周报的模式一致：先列待处理项 → 逐项分析 → 自动刷新。

## Risks / Trade-offs

- **[R] 自适应阈值在极端群聊中可能失效**：如果群聊是持续不间断的（如 24 小时技术支持群），可能永远找不到间隙 → 整个尖峰时段变成一个巨大的事件组 → **Mitigation**：上限 500 条强制切分；活跃群判定阈值可调
- **[R] 串行处理慢**：50 个窗口可能需要 10-15 分钟 → **Mitigation**：用户可随时取消；窗口按时间倒序排列，最近的先分析；未来可在设置中开放"轻度并发(2)"选项
- **[R] event_windows 表增长**：每次重新检测都新增窗口 → **Mitigation**：重新检测时删除同群同时段旧窗口；定期清理策略可在后续版本加入
- **[R] Breaking 前端**：`POST /detect` 返回格式变化 → **Mitigation**：前端同步更新；v1.18.0 用户量小

## Migration Plan

1. 数据库迁移：新增 `event_windows` 表 + `events.window_id` 列（`ALTER TABLE ADD COLUMN`）
2. 后端：新增 `routers/event_windows.py`；修改 `services/event_detector.py`（自适应切分）；修改 `routers/events.py`（移除内联分析）
3. 前端：修改 `Events.vue` 两步交互；修改 `EventTimeline.vue` 展示窗口卡片；新增 API 调用
4. 部署：代码替换，自动迁移，无数据迁移需求（v1.18.0 的 events 数据保持兼容）
5. 回滚：数据库 `event_windows` 表不影响 events 查询；旧版前端仍可调用 `GET /events` 查看已有事件
